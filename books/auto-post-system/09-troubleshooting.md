---
title: "09 トラブルシュート"
---

よくある問題と対処法をまとめます。

例:

- Secrets が未設定でワークフローが失敗する
  - チェック項目と再設定方法
- LLM のレスポンスが不正（フォーマット違反）
  - 入力プロンプトの見直しとリトライ方針
- 投稿が成功したが表示されない
  - X API の権限や Rate limit 確認

各項目に再現手順とログの抜き出し方を記載してください。

この章は運用で遭遇しやすい障害ごとに、優先度付きの初動対応と原因切り分けを示します。

1. ワークフローが Secrets 未設定で失敗する

症状: GitHub Actions ログに `Missing secrets` や `401` が出る

初動対応:

- Actions の実行履歴から該当ワークフローを開き、`env` / `secrets` の設定を確認
- リポジトリの `Settings > Secrets and variables > Actions` に必要なキーが存在するか確認
- 本番環境 (Environments) に制限がかかっていないかを確認

再発防止:

- 必須シークレットがない場合に失敗する前にチェックするステップをワークフローに入れる

2. Gemini のレスポンスがフォーマット違反や不正な JSON を返す

症状: パースエラー、期待キーがない、生成本文が不適切

初動対応:

- 実行ログ（レスポンスボディ）を取得し、該当の出力を確認
- 生成時のプロンプトとパラメータ（temperature, max_tokens）をログと照合

切り分け手順:

- 同じプロンプトを手動で API に投げて再現性を確認
- 必要ならフォールバックで再生成を試す（回数上限を設定）

3. 投稿が成功したがコンテンツが見えない／消えた

症状: 投稿 API は 200/201 を返すが、UI に表示されない

原因候補:

- 投稿に対する表示制限（アカウントの一時停止、投稿の保留）
- メディアアップロードに失敗しており、投稿が部分的に失敗している
- API 側の整合性の遅延

対応:

- 投稿ID をログから取得して API で詳細を問い合わせる
- 投稿を削除して再投稿する手順を用意する（必要に応じて）

4. Rate limit / 429 が頻発する

対応:

- リトライロジック: 指数バックオフを実装する
- バッチ化: リクエストをまとめて送る、頻度を下げる
- 重要なジョブはオフピークに移動する

5. ストレージ（posted.json）が壊れている／読み書きエラー

対応:

- まずバックアップ（`data/posted.json.bak` を作る）
- JSON の整合性を `jq` でチェック:

```bash
jq . data/posted.json > /dev/null
```

- 破損している場合は最新のバックアップから復元、またはロールフォワード用のログから再構築する

6. 同時実行での重複投稿

対応:

- まずは `concurrency` 設定で同時実行を防ぐ
- 永続 DB（UNIQUE インデックス）に移行し、書き込み時に失敗したら重複として扱う

ログの抜き出し方（Tips）

- GitHub Actions: 実行結果の `Artifact` や `Logs` をダウンロードして調査
- ローカル: `logs/last-run.json` を開き、`traceId` でフィルタする
- `jq` を使った検索例:

```bash
jq 'select(.traceId=="<id>")' logs/events.log
```

エスカレーションと Runbook

- 重大インシデントの場合は、オンコールに連絡し、影響範囲を評価する
- Runbook には「すぐにやること」「ログの集め方」「一時的な回避策」「恒久対応」を記載する

次の作業: 校正と最終確認（目次にファイル名が表示されないことの確認を含む）を行います。
